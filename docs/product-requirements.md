# 产品需求文档 (PRD)：LLM-Eval-Kit

> 版本：1.0 | 状态：MVP 已发布 | 负责人：AI 工程化团队

---

## 1. 概述

### 1.1 产品定位

**一句话**：面向中文业务场景的轻量级 LLM 效果量化评测工具箱。

**目标用户画像**：

| 用户类型 | 核心诉求 | 使用场景 |
|----------|----------|----------|
| AI 应用开发者 | 快速对比模型效果 | Prompt 迭代、模型选型 |
| 中小团队技术负责人 | 控制 API 成本 | 成本效益分析、预算规划 |
| 独立开发者 | 零成本验证想法 | 前期技术调研、Demo 展示 |
| 技术面试候选人 | 作品集 / 代表作 | 展示系统设计能力、工程化水平 |

### 1.2 用户痛点

1. **凭感觉评估**："这个模型好像回答得好一些"——没有量化指标
2. **工具太重**：全链路评测框架需要 torch/transformers/GPU，个人项目负担过重
3. **成本不可见**：API 调用费用月底才看到账单，缺乏实时成本感知
4. **回归无感知**："改了 Prompt 好像变差了？"——没有版本对比
5. **报告太复杂**：学术 benchmark 的指标对业务场景参考价值有限

### 1.3 核心假设

| 假设 | 验证方法 | 风险等级 |
|------|----------|----------|
| 开发者愿意用 CLI 而非 Web UI 做评测 | GitHub Star / Issue 反馈 | 中 |
| 规则评分对业务场景有参考价值 | 与人工评分的相关性分析 | 高 |
| 双币种成本模型能降低用户选择成本 | 用户留存率 | 低 |

## 3. 用户场景

### 场景 1：Prompt 迭代验证

```
角色：AI 应用开发者（小明）
目标：验证新 Prompt 是否比旧 Prompt 效果好
操作：
  1. 准备 10 条测试问题（JSONL）
  2. 分别用旧 Prompt 和新 Prompt 各跑一次评测
  3. 对比综合评分
  4. 如果评分提升 → 上线新 Prompt
验收：全程 < 5 分钟，无需人工抽测
```

### 场景 2：模型选型决策

```
角色：技术负责人（老王）
目标：在 DeepSeek、千问、GPT 之间做选择
操作：
  1. 准备 30 条业务测试问题
  2. 配置三个模型的 API Key
  3. 运行 llm-eval-kit eval 一次评测三个模型
  4. 查看对比表和成本效益分析
  5. 决定：质量优先选 GPT，性价比优先选 DeepSeek
验收：一份报告给决策，数据说话而非直觉
```

### 场景 3：CI 质量门禁

```
角色：DevOps 工程师（小李）
目标：每次 Prompt 改动自动检测质量是否下降
操作：
  1. GitHub Actions 中集成 llm-eval-kit
  2. PR 时自动运行评测
  3. 与 baseline 对比
  4. 评分下降 > 5% 则阻断合并
验收：无人工介入，自动门禁
```

## 4. 非功能需求

### 4.1 性能指标

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 评分响应 | < 10ms / sample | 规则评分 |
| 报告生成 | < 1s / 100 samples | 终端 + HTML |
| 并发支持 | 10+ 并发 | 测试环境 |
| 启动时间 | < 1s | pip install 后首次运行 |

### 4.2 兼容性

| 维度 | 要求 |
|------|------|
| Python | ≥ 3.8 |
| OS | Windows / macOS / Linux |
| 终端 | UTF-8（GBK 降级使用 "CNY" 前缀） |
| API 接口 | OpenAI 兼容格式 |

### 4.3 安全约束

- API Key 通过参数传入，不落盘
- 不上传任何数据到第三方服务
- 无遥测 / 匿名统计

## 6. 竞品分析

| 产品 | 定位 | 依赖 | 优势 | 劣势 |
|------|------|------|------|------|
| **llm-eval-kit** | 中文轻量业务评测 | httpx + jieba | 轻量、开箱即用、业务适配 | 功能有限 |
| LangSmith | 全链路 LLM 可观测 | 重 | 功能全面、可视化 | 太重、付费 |
| DeepEval | 单元测试式评测 | 重 | CI 集成、断言式 | 英语优先 |
| lm-evaluation-harness | 学术 benchmark | torch + transformers | 学术标准 | 太大、业务不相关 |
| 人工抽测 | 无 | 无 | 灵活 | 不量化、不可复现 |

**差异化定位**：不做大而全，专注中文业务场景"效果—成本—效率"三个核心指标，3 分钟出结论。

## 7. 风险评估

| 风险 | 概率 | 影响 | 缓解策略 |
|------|------|------|----------|
| 规则评分与用户感知不一致 | 中 | 高 | 提供 LLM Judge 作为备选；开放自定义权重 |
| API Key 泄露 | 低 | 高 | 明确文档说明安全最佳实践 |
| 用户不愿意装 CLI 工具 | 中 | 中 | 保留 Python API 方式，三行代码可运行 |
| LLM Judge 成本过高 | 中 | 中 | 默认规则评分，Judge 为可选 |
