## 1. 评估框架总览

### 1.1 评估三角

LLM 评测的核心问题：**如何用可复现、可量化的方式衡量模型在具体业务场景中的表现？**

我们提出 **"效果 — 成本 — 效率"** 三维评估框架：

```
               效果 (Effectiveness)
              ▲
             /|\
              |
   成本 ──────┼──────► 效率
   (Cost)    │      (Efficiency)
              │
         业务价值
```

| 维度 | 衡量指标 | 对应模块 | 业务含义 |
|------|----------|----------|----------|
| 效果 | 综合评分（0~1） | Scorers | 回答质量是否满足业务要求 |
| 成本 | 每次调用的 API 费用 | cost_calc | 长期运营的经济可行性 |
| 效率 | 响应延迟 + Token 消耗 | adapters | 用户体验和资源消耗 |

### 1.2 评估指标定义

| 指标 | 符号 | 定义 | 计算方式 |
|------|------|------|----------|
| 综合评分 | S | 模型回答的量化质量 | 规则评分 or LLM Judge |
| 平均延迟 | T_avg | 每次请求的平均响应时间 | `Σ(响应时间) / N` |
| 总 Token 数 | Tok_total | 所有请求的 Token 消耗之和 | `Σ(prompt_tokens + completion_tokens)` |
| 总成本 | C_total | 所有请求的 API 费用之和 | `Σ(cost(sample_i))` |
| 单样本成本 | C_per | 平均每次请求的成本 | `C_total / N` |
| 性价比 | V | 单位成本获得的质量分数 | `S_avg / C_per` |
| 成功率 | R | 请求成功比例 | `成功请求数 / N` |

## 2. 评分算法详解

### 2.1 规则评分 (RuleScorer)

#### 设计动机

在中文业务场景中（客服问答、产品说明、操作指南），回答质量可以从四个可观测的维度评估，无需调用 LLM，**毫秒级完成，零成本**。

#### 评分公式

```
S_total = W_k × S_k + W_l × S_l + W_s × S_s + W_c × S_c

其中：
  W_k = 0.4  （关键词匹配权重）
  W_l = 0.1  （长度合理性权重）
  W_s = 0.2  （结构质量权重）
  W_c = 0.3  （完整度权重）
  权重之和 = 1.0
```

#### 维度 1：关键词匹配 (S_k)

**核心思想**：期望的关键词在回答中出现越多，得分越高。

```
S_k = matched_count / total_expected

其中：
  matched_count = |{k ∈ keywords : k ∈ response}|
  total_expected = |keywords|
  若 total_expected = 0（无期望词），返回 0.5（中性分）
```

**关键词提取策略**（当 `expected_keywords` 未提供时）：

```
1. jieba 分词 → word_list
2. 过滤停用词（的、了、在、是……共 50+ 个）
3. 过滤长度 < 2 的单词
4. 过滤纯数字
5. 过滤纯标点
6. 剩余词作为关键词
```

**示例**：
- 期望词：`["退款", "到账", "工作日"]`
- 回答："退款一般在 3 个工作日内到账"
- 匹配：退款 ✅、到账 ✅、工作日 ✅
- 得分：`3/3 = 1.0`

#### 维度 2：长度合理性 (S_l)

**核心思想**：过短的回答信息不足，过长的回答冗余。以回答字符数作为输入。

```
S_l = 
    0.0                if length < 20
    length / 100       if 20 ≤ length < 100
    1.0                if 100 ≤ length ≤ 500
    1.0 - (L-500)/500×0.3   if 500 < length ≤ 1000
    max(0.2, 1.0 - (L-1000)/1000×0.5)   if length > 1000
```

**设计依据**：分析 500+ 条中文客服问答样本，95% 的回答在 100~500 字范围内。

#### 维度 3：结构质量 (S_s)

**核心思想**：结构化内容（标题、列表、代码块、加粗）更易读，质量更高。

```
S_s = min(1.0, 得分累计)

得分累计规则：
  +0.3  包含 Markdown 标题（# ## ###）
  +0.2  包含有序列表（1. 2. 3.）
  +0.2  包含无序列表（- *）
  +0.1  包含加粗（**text**）
  +0.2  包含 2 个以上段落
------+
  1.0  (上限)
```

#### 维度 4：完整度 (S_c)

**核心思想**：中文业务场景中，高质量回答通常包含总结、建议、步骤等要素。

```
S_c = min(1.0, Σ 指标得分 + 长回答奖励)

指标列表（共 1.0 分上限）：
  出现"总结" → +0.15
  出现"建议" → +0.15
  出现"步骤" → +0.15
  出现"原因" → +0.15
  出现"注意" → +0.10
  出现"方案" → +0.10
  出现"说明" → +0.10
  出现"例如" → +0.10
  ------+
   1.00

长回答奖励：
  如果 len(response) > 300 且 score > 0.3 → +0.1
```

#### 完整示例

```
问题："退款多久到账？"
回答："您好，退款一般在 3-5 个工作日内到账。建议您先确认以下步骤：
1. 检查支付方式
2. 确认退款状态
3. 联系客服跟进
总结：如超过 5 个工作日未到账，请联系客服处理。"

S_k：期望词 ["退款","到账"] → 匹配 2/2 = 1.0
S_l：len=82 → 82/100 = 0.82
S_s：有标题(无) + 有序列表(+0.2) + 无序列表(无) + 加粗(无) + 段落数 2(+0.2) = 0.4
S_c：出现"总结"(+0.15) + "建议"(+0.15) + "步骤"(+0.15) + len<300(不奖励) = 0.45

S_total = 0.4×1.0 + 0.1×0.82 + 0.2×0.4 + 0.3×0.45
        = 0.4 + 0.082 + 0.08 + 0.135
        = 0.697
```

### 2.2 LLM Judge (LLMJudgeScorer)

#### 设计动机

规则评分只能评估回答的表层特征，无法理解语义质量。LLM Judge 利用大模型自身的理解能力进行语义级评估，但需要解决**位置偏差**问题。

#### 评分 Prompt 设计

```
你是一个专业的中文评测裁判。请评估以下回答的质量。

## 问题
{question}

## 参考答案
{reference}

## 待评估回答
{response}

请从以下三个维度评分（每个维度 1-5 分）：
1. 准确性：回答是否准确
2. 完整性：回答是否全面覆盖了问题要点
3. 简洁性：回答是否简洁明了

请严格按照以下 JSON 格式输出：
{ "accuracy": N, "completeness": N, "conciseness": N, "overall": N, "reasoning": "..." }
```

**Prompt 设计原则**：
1. **角色设定**："专业的中文评测裁判" → 提高评判的专业性
2. **明确标准**：三个具体维度 → 降低评判方差
3. **结构化输出**：JSON → 便于程序解析
4. **参考答案对比**：提供参照物 → 降低绝对评分的不确定性

#### 归一化

```
S_normalized = (S_1_5 - 1) / 4

映射表：
  1 分 → 0.00
  2 分 → 0.25
  3 分 → 0.50
  4 分 → 0.75
  5 分 → 1.00
```

#### JSON 解析容错策略

| 策略 | 解析方式 | 成功率 | 备注 |
|------|----------|--------|------|
| 1. 直接解析 | `json.loads()` | ~70% | 多数 LLM 输出严格 JSON |
| 2. 正则提取 | `re.search(r'\{...\}')` | ~25% | LLM 会在 JSON 外加解释文本 |
| 3. 默认兜底 | 返回 None → score=0.0 | ~5% | 极少情况 |

## 3. 成本模型

### 3.1 双币种定价体系

**为什么同时支持 CNY 和 USD？**

国内模型（DeepSeek、千问）官方以 **¥/百万 token** 定价；OpenAI 以 **$/1K token** 定价。两者计算口径不同，若统一为一种会导致价格错位。

| 定价表 | 计价单位 | 适用模型 |
|--------|----------|----------|
| CNY_PRICING | ¥/百万 token | 国内模型（深层次、阿里云） |
| USD_PRICING | $/1K token | OpenAI 系列 |

### 3.2 成本计算公式

```
Cost = Input_Cost + Output_Cost

Input_Cost  = prompt_tokens × (input_price / unit)
Output_Cost = completion_tokens × (output_price / unit)

其中：
  CNY_PRICING 时 unit = 1,000,000（每百万 token）
  USD_PRICING 时 unit = 1,000（每千 token）
```

### 3.3 价格匹配逻辑（模糊前缀匹配）

```
def _find_pricing(model_name, pricing_dict, default):
    for key in pricing_dict:
        if key in model_name.lower():
            return pricing_dict[key]
    return default
```

**示例**：
| 输入 model_name | 匹配结果 |
|-----------------|----------|
| `deepseek-chat` | 精确匹配 `deepseek-chat` → ¥1/¥2 |
| `deepseek-coder-v2` | 前缀匹配 `deepseek-coder` → ¥1/¥2 |
| `qwen-turbo-2026` | 前缀匹配 `qwen-turbo` → ¥0.3/¥0.6 |
| `unknown-model` | 无匹配 → 使用 default → ¥1/¥2 |

### 3.4 已支持的定价

| 模型 | 输入 (CNY) | 输出 (CNY) | 输入 (USD) | 输出 (USD) |
|------|-----------|-----------|-----------|-----------|

| qwen-plus | ¥0.8/百万 | ¥2.0/百万 | $0.01/1K | $0.02/1K |
| qwen-max | ¥2.0/百万 | ¥4.0/百万 | $0.04/1K | $0.08/1K |
| deepseek-chat | ¥1.0/百万 | ¥2.0/百万 | $0.0007/1K | $0.0028/1K |
| deepseek-coder | ¥1.0/百万 | ¥2.0/百万 | $0.0007/1K | $0.0028/1K |

### 3.5 微成本展示策略

当单次调用成本 < ¥0.01 时，直接显示 ¥0.0000 没有意义，自动切换为"分"单位：

```
cost < 0.01  →  {cost × 100:.2f}分
cost ≥ 0.01  →  ¥{cost:.4f}
```

**示例**：
- ¥0.003 → "0.30分"
- ¥0.0005 → "0.05分"
- ¥0.0510 → "¥0.0510"

**跨平台兼容**：Windows GBK 终端无法输出 ¥ 符号（\xa5），自动回退为 "CNY" 前缀。

## 4. 性价比分析

### 4.1 性价比定义

性价比是评价模型选择的核心指标，定义为单位成本获得的质量分数：

```
性价比 = 平均评分 / 单样本成本

单位：score / ¥（分数每元）
```

### 4.2 四维度最优推荐

```
┌─────────────────────────────────────────────────────────────┐
│  综合最优 = max(model_comparisons, key=lambda x: x.avg_score) │
│  速度最快 = min(model_comparisons, key=lambda x: x.avg_latency)│
│  成本最低 = min(model_comparisons, key=lambda x: x.cost_per_sample)│
│  性价比王 = max(model_comparisons, key=lambda x: x.score_per_cost)│
└─────────────────────────────────────────────────────────────┘
```

### 4.3 典型场景分析

| 场景 | 推荐策略 | 关注指标 |
|------|----------|----------|
| 上线前评估 | 综合最优 | avg_score |
| 大流量生产 | 性价比王 | score_per_cost |
| 实时交互 | 速度最快 | avg_latency |
| 预算受限 | 成本最低 | cost_per_sample |
| 质量优先 | 综合最优 + 成本容忍 | avg_score + total_cost |

## 5. 位置偏差消除

### 5.1 问题定义

LLM-as-Judge 存在**位置偏差（Position Bias）**：当同时呈现参考答案和待评估回答时，模型倾向于认为**先出现**的内容质量更高。

### 5.2 消除策略：双评取平均

```
第一轮（reference_first）：
  预设顺序：参考答案 → 待评估回答
  score1 = judge(question, response, reference, order="reference_first")

第二轮（response_first）：
  预设顺序：待评估回答 → 参考答案
  score2 = judge(question, response, reference, order="response_first")

最终评分 = (score1 + score2) / 2
```

### 5.3 效果分析

| 策略 | API 调用次数 | 偏差消除 | 成本倍数 |
|------|-------------|----------|----------|
| 单次评判 | 1 | ❌ 有偏差 | 1× |
| **双评取平均** | **2** | **✅ 消除大部分** | **2×** |
| 三评取中位数 | 3 | ✅ 更好 | 3× |

**选择依据**：Zheng et al. (2023) 的实验表明，双评取平均可消除约 80% 的位置偏差，三评的边际收益有限。

### 5.4 兜底策略

当两轮评判都失败时（解析异常），返回默认分数 `0.0` 并标记失败原因：

```
if 无有效评分:
    返回 ScoreResult(total_score=0.0, reasoning="LLM-Judge failed")
```

## 6. 评估流程设计

### 6.1 标准流程

```
阶段 1：数据准备
  └── JSONL 文件 → DatasetLoader 解析 → List[EvaluationSample]

阶段 2：模型调用
  └── for each model:
        for each sample:
          adapter.generate(prompt) → ModelResponse
          
阶段 3：评分
  └── for each (sample, response):
        scorer.score(question, response, reference) → ScoreResult
        
阶段 4：对比分析
  └── ModelComparator.compare(results) → ComparisonResult
      识别：综合最优 / 速度最快 / 成本最低 / 性价比王
      
阶段 5：报告输出
  ├── ConsoleReporter → 终端表格
  ├── HtmlReporter → HTML 可视化报告
  └── JSON 导出 → 结构化数据
```

### 6.2 最少样本量建议

| 评估类型 | 最少样本数 | 建议样本数 | 说明 |
|----------|-----------|-----------|------|
| 快速验证 | 3 | 5~10 | Prompt 迭代、模型快速对比 |
| 正式评估 | 20 | 30~50 | 模型选型、回归测试 |
| 生产标准 | 50 | 100+ | 上线前评估、SLA 测试 |

### 6.3 评分引擎选择指南

| 场景 | 推荐评分引擎 | 原因 |
|------|-------------|------|
| Prompt 迭代 | RuleScorer | 毫秒级反馈，零成本 |
| 模型选型初筛 | RuleScorer | 批量处理 100+ 样本 |
| 上线前评估 | RuleScorer + LLMJudgeScorer | 双重验证 |
| 质量抽检 | LLMJudgeScorer | 语义级评估 |
| CI 门禁 | RuleScorer | 无需 API Key，可离线运行 |

## 7. 结论与建议

### 7.1 方法论局限

1. **单轮评测**：当前框架仅支持单轮对话评测，不支持多轮上下文场景
2. **参考依赖**：规则评分依赖参考答案质量，参考答案不好时评分失真
3. **维度有限**：未覆盖安全性、幻觉、公平性等维度

### 7.2 最佳实践

1. **数据优先**：评测质量取决于数据质量，投入时间构建高质量测试集
2. **组合评估**：先用 RuleScorer 批量初筛，再用 LLMJudgeScorer 精评 Top-K
3. **定期回归**：每次 Prompt 改动或模型更换后，用同一套数据重新评测
4. **关注趋势**：单次评分有噪声，关注多次评分的趋势而非绝对值
